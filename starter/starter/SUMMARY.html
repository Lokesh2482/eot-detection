End-of-Turn Detection - Submission Summary

Overview

This submission detects end-of-turn from causal acoustic features only, using no pretrained models and CPU-only training/inference. It beats the silence-only baseline (~1600ms mean response delay at 5% interrupted turns) by a wide margin on both provided languages.

Approach

1. Feature engineering (56 causal features)

  • Multi-scale energy: level, variability, slope, and start/end change over 5 trailing windows (0.1s-1.5s)
  • Final-segment energy relative to the speaker's own recent baseline (median/IQR normalized)
  • Active-speech run length and silence-since-last-speech
  • Speaker-relative F0 (pitch) in semitones: slope, range, and deviation from the speaker's own trailing baseline, across the same multi-scale windows
  • Voicing fraction and last-voiced-run timing
  • Spectral centroid slope, flatness, flux, and zero-crossing rate over 0.5s/1.5s windows

All features are computed strictly from audio before each pause's pause_start - never pause_end or any future audio. This was verified directly in the feature code.

2. Model

  • A randomized tree ensemble (ExtraTrees-style), implemented from scratch in NumPy - no scikit-learn/XGBoost/LightGBM/deep learning dependency for the model itself
  • 300 trees, max depth 6, class-balanced via sample weighting
  • Turn-grouped 5-fold cross-validation during development, so no turn's pauses appear in both train and validation within a fold
  • Trained on English + Hindi combined, since the hidden test set is "mostly Hindi" and shared prosodic signal was expected to generalize better than training on Hindi alone - this was tested directly (see Results) and confirmed
  • Final model refit on all 496 available rows and saved as model.pkl for inference

3. Inference

predict.py --data_dir <folder> --out predictions.csv loads the saved model and extracts features per pause - it performs no retraining, matching the requirement that it run on a previously unseen folder with the same schema.

The combined model was chosen for submission: it produces a lower operational delay on Hindi despite the Hindi-only model having higher raw AUC, likely because the combined model benefits from roughly double the training data for the tree ensemble, and the scorer optimizes a specific threshold/delay operating point rather than overall ranking quality.

What was done by the coding agent vs. by hand

  • Coding agent: initial causal feature extraction module (energy/F0/spectral features across multiple windows), the turn-grouped cross-validation harness, the from-scratch NumPy tree ensemble (needed since scikit-learn was unavailable in the execution environment), and the predict.py inference script.
  • Done by hand / directed: restructuring the training script to train multilingually rather than per-language; running and interpreting the Hindi-only vs. combined comparison to make an evidence-based training decision rather than assuming multilingual training would win; separating shared model classes into ensemble.py to fix a pickle/import bug that broke predict.py; distinguishing and correctly labeling OOF vs. in-sample evaluation numbers rather than reporting only the more flattering in-sample score; feature importance review to identify near-zero-value features (context_s, context_is_short, f0_reference_available) as pruning candidates.

Why this beats the status quo

The silence-only baseline (naive VAD-style endpointing) waits a fixed timeout regardless of what was actually said, averaging ~1600ms at a 5% interruption budget. This system instead reads the trajectory of energy, pitch, and spectral shape in the last 0.1-1.5 seconds of speech to predict turn completion, cutting the honest (OOF) mean response delay to 826ms on Hindi and 1234ms on English - roughly a 23-48% reduction in how long the agent keeps users waiting, at the same 5% false-interruption budget.

Competition Compliance

  • Strictly causal (verified in code): only audio before pause_start is used
  • CPU-only training and inference
  • No pretrained models or downloaded weights
  • No external datasets
  • Allowed libraries only: NumPy (model and features), Python stdlib (wave, csv)
  • predict.py performs inference only, no retraining
